# Lessons

- When adjusting fixed-layout canvas UI, define explicit inner padding and column widths for each section before moving buttons or text. Hard-left origins and hand-tuned offsets look aligned in code and still render visibly off-center.
- When centering fixed-layout sidebar controls, center each control row against the sidebar frame itself and add a regression test for the resulting left/right margins. Do not assume matching widths will look centered without measuring them.
- Do not introduce new text or other visible UI elements without explicit user instruction. Prefer moving or removing existing copy over inventing new guidance text.
- In gameplay and play-test views, keep the sidebar focused on controls unless the user explicitly asks for instructional copy there. New obstacle or mechanic hints should not appear as visible sidebar text by default.
- When a Phaser scene is reused across launches, reset transient control state in `init` instead of relying on class field defaults. Play-test regressions like missing selected outlines can come from stale `tool`, `brushIndex`, hover, or accumulator state carrying across runs.
- In fixed-layout canvas HUDs and popups, right-align numeric values away from adjacent buttons and size modal boxes from their longest prompt text first. Add layout tests for the numeric/button gap and popup text bounds instead of relying on screenshots alone.
- For pixel-art UI text, do not fall back to generic system fonts if readability matters. Use a self-hosted readable pixel font, wait for it before boot, and keep the font family centralized so future scenes cannot silently drift back to less legible defaults.
- Pixel-font readability is not solved by aesthetic alignment alone. After swapping fonts, validate dense glyph differentiation (`H`, `W`, `M`, numerals) in-browser and keep a global size-step helper so the whole UI can move up one readable notch without hand-editing every scene.
- When evaluating replacement fonts, do not stop at titles or empty screens. Validate the densest UI surfaces first, especially level cards and HUD labels. A font that looks fine on the splash screen can still fail on card grids or compact sidebars.
- If a font improves glyph separation but widens dense screens too much, prefer the narrower retro candidate instead of forcing layout exceptions. Font choice should simplify the UI, not create per-screen compensating fixes.
- When the chosen font family is correct but still reads too small, change the global size step in one place instead of tweaking individual scenes. Then re-check constrained screens specifically for overflow, because size regressions will show up there first.
- In fixed-size pixel UI controls, centered text is not enough. Large numeric/button labels need explicit internal x/y padding and a font-size cap so glyphs do not clip against the top-left edge after a font change.
- In Phaser UI, selection outlines and modal overlays need explicit draw order. If a stateful border or dialog should sit on top, encode the layer order in a shared helper and test it instead of assuming creation order will hold after refactors.
- If a selected-state outline has regressed more than once, stop relying on visual memory. Lock both the layer order and the outline thickness/inset in pure tests so refactors cannot silently weaken or bury the selected state again.
- If a HUD meter is not self-explanatory, label the meter itself instead of relying on nearby labels or user inference. Put the label on the shared meter layout so it stays centered and test that anchor.
- For pixel-font labels inside framed bars or buttons, geometric center can still look low. Encode an optical center offset in the shared layout helper and test that it stays slightly above the raw midpoint.
- As the game evolves, update the durable repo context in the same pass as the code change. Keep the README, `docs/` knowledge base, and code-level intent comments aligned with shipped behavior, design decisions, regressions, and future direction so that context does not get stranded in chat history.
- In fixed-layout canvas UI, elements that live on the same plane should never visually touch once shadows and strokes are included. Measure the full rendered footprint, not just the raw layout box, and lock a small gap in tests so buttons cannot graze panel or section borders.
- When adding CJK localization to a fixed pixel UI, do not treat translation as a text-only change. Chinese glyph metrics are usually taller and denser than the Latin font they replace, so shared buttons and compact HUD/editor labels need locale-aware font sizing, weight, and fit rules before you start hand-tuning individual strings.
- For procedural content, do not scale the shipped campaign count until the generation method itself has been vetted on a small sample. Generate a review pack first, prove the layouts do not collapse into obvious left-to-right lanes, and only then expand the count.
- When a user says generated maps feel "AI", treat that as a structure problem, not just a theme problem. Add spatial constraints that reward scattered anchors, mixed axes, and freeform terrain blobs instead of polishing the same linear topology.
- If live gameplay is intentionally probabilistic, do not treat one lucky replay as proof that a generated level is fair. Budget and rank levels against multi-seed success rate so randomness does not turn into fake difficulty.
- For procedural difficulty tuning, use replay success as the fairness gate and target pressure as the tightness floor. Grow budgets until the fairness gate is met, then tighten back down; otherwise the generator oscillates between unfairly tight and wildly overgenerous levels.
- When generation scales up from a review pack to a campaign, keep tests aligned to the actual product ask rather than the last tuning experiment. Over-constrained regression rules can silently force tutorial levels to behave like mastery levels and blow up generation time.
